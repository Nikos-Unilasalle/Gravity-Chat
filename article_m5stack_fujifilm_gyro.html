Stabilisation gyroscopique en post-prod avec le M5StickC et le Fujifilm X-M5

Le Fujifilm X-M5 n'embarque pas de métadonnées gyroscopiques dans ses fichiers vidéo. Ce guide montre comment pallier ce manque avec un M5StickC Pro fixé sur la griffe flash, en enregistrant les données IMU synchronisées, puis en les injectant dans Gyroflow pour une stabilisation optique de haute qualité.

Niveau: Avancé Durée de lecture: ~18 min Outils: Arduino IDE · Gyroflow · Python 3.11 Matériel: M5StickC Plus2 · X-M5

1. Pourquoi le X-M5 pose problème

Le Fujifilm X-M5 est une caméra remarquable pour son format compact et son rendu couleur. Mais contrairement au X-H2S ou au Sony A7S III, il n'enregistre aucune donnée de mouvement dans ses flux vidéo MOV ou MP4. Pas de flux gyroscopique embarqué, pas de métadonnées OIS exploitables — ce que Gyroflow, Reelsteady ou Warp Stabilizer peuvent utiliser directement.

La stabilisation logicielle sans données de mouvement repose sur une analyse optique du flux pixel par pixel (algorithmes de type Lucas-Kanade ou RAFT). Elle est coûteuse en calcul, imprécise sur les mouvements rapides, et détruit les bords du cadre. La solution propre : enregistrer soi-même les données IMU en externe, synchronisées avec la vidéo, puis les fournir à Gyroflow comme source de vérité.

Principe fondamental Gyroflow peut ingérer un fichier CSV ou GCSV contenant des données gyroscopiques horodatées (timestamps en µs, rotations en dps), à condition que la fréquence d'échantillonnage soit suffisamment élevée — idéalement 200 Hz ou plus — et que la synchronisation temporelle soit gérée avec précision.

2. Anatomie du M5StickC Plus2

Le M5StickC Plus2 embarque un IMU MPU6886 — un combo accéléromètre 3 axes + gyroscope 3 axes d'InvenSense. Ses caractéristiques clés pour notre usage :

Paramètre Valeur Note
ODR gyroscope max 1 kHz Via registre SMPLRT_DIV
Plage gyro (configurable) ±250 / 500 / 1000 / 2000 dps ±1000 dps recommandé pour le handheld
Résolution ADC 16 bits Sensibilité 32.8 LSB/dps à ±1000 dps
Interface I²C (400 kHz) ou SPI M5Stack utilise I²C par défaut
Horloge interne PLL sur gyro X Meilleure stabilité que oscil. interne

L'ESP32 du M5StickC dispose d'un timer hardware 64 bits capable d'une précision sub-microseconde. C'est ce timer que nous utiliserons pour horodater chaque mesure, et non millis() qui présente une dérive notable sur de longues séquences.

3. Firmware : enregistrement IMU haute fréquence

Stratégie de sampling

Deux approches existent : le polling bloquant dans loop(), ou l'utilisation du registre FIFO du MPU6886. Le FIFO est préférable : il découple le rythme de lecture du rythme de production des données, évite les skips lors des pics CPU (écriture SD, affichage LCD).

firmware/imu_logger.ino — configuration MPU6886
// Registres MPU6886 (adresse I²C 0x68)
#define MPU6886_ADDR        0x68
#define REG_SMPLRT_DIV      0x19  // diviseur de taux d'échantillonnage
#define REG_CONFIG          0x1A  // filtre passe-bas numérique (DLPF)
#define REG_GYRO_CONFIG     0x1B  // plage gyroscope
#define REG_FIFO_EN         0x23  // activation FIFO
#define REG_USER_CTRL       0x6A  // contrôle FIFO
#define REG_FIFO_COUNTH     0x72
#define REG_FIFO_R_W        0x74

void initIMU() {
  Wire.begin(32, 33);          // SDA=GPIO32, SCL=GPIO33 sur M5StickC Plus2
  Wire.setClock(400000);

  writeReg(REG_SMPLRT_DIV, 0x04);  // ODR = 1000 / (1+4) = 200 Hz
  writeReg(REG_CONFIG,     0x03);  // DLPF_CFG=3 : 41 Hz BW, 5.9ms delay
  writeReg(REG_GYRO_CONFIG,0x10);  // FS_SEL=2 → ±1000 dps
  
  // Activer FIFO pour gyro uniquement (bits XG, YG, ZG)
  writeReg(REG_FIFO_EN,    0x70);
  writeReg(REG_USER_CTRL,  0x40);  // FIFO_EN bit
}
Choix du DLPF Le DLPF_CFG=3 applique un filtre passe-bas à 41 Hz sur le gyroscope. C'est un compromis : on filtre le bruit haute fréquence (vibrations moteur, bruit thermique) sans retarder excessivement les données (5.9 ms de latence de groupe). Pour une caméra portée à la main, 41 Hz est suffisant — les mouvements humains n'excèdent pas 10–15 Hz.

Lecture FIFO et horodatage

// Timer hardware ESP32 — précision ~1 µs
hw_timer_t* timer = NULL;
volatile uint64_t isr_timestamp = 0;

void IRAM_ATTR onTimer() {
  isr_timestamp = esp_timer_get_time();  // µs depuis boot
}

struct ImuSample {
  uint64_t ts_us;   // timestamp en microsecondes
  float    gx, gy, gz;  // deg/s
  float    ax, ay, az;  // g
};

void readFifo(std::vector<ImuSample>& buf) {
  uint16_t count = (readReg(REG_FIFO_COUNTH) << 8) | readReg(REG_FIFO_COUNTH + 1);
  uint8_t  samples = count / 6;  // 6 bytes par sample gyro (3 axes × 2 bytes)
  
  // Timestamp du dernier sample dans la fenêtre
  uint64_t now = esp_timer_get_time();
  uint64_t dt  = 1000000 / 200;  // 5000 µs à 200 Hz

  for (int i = 0; i < samples; i++) {
    int16_t rawGx = readFifoWord();
    int16_t rawGy = readFifoWord();
    int16_t rawGz = readFifoWord();

    ImuSample s;
    s.ts_us = now - (uint64_t)(samples - i - 1) * dt;
    s.gx = rawGx / 32.8f;   // sensibilité ±1000 dps
    s.gy = rawGy / 32.8f;
    s.gz = rawGz / 32.8f;
    buf.push_back(s);
  }
}

Écriture sur carte SD en GCSV

Gyroflow supporte nativement le format GCSV (Gyroflow CSV), un format texte simple avec un en-tête déclarant les axes et l'unité. On écrit par blocs de 512 bytes pour minimiser les appels I/O à la carte SD et éviter les micro-latences qui décaleraient le thread d'acquisition. 

void writeGCSV(File& f, std::vector<ImuSample>& buf) {
  String block = "";
  for (auto& s : buf) {
    block += String(s.ts_us);       // colonne 1 : µs
    block += "," + String(s.gx, 4);
    block += "," + String(s.gy, 4);
    block += "," + String(s.gz, 4);
    block += "," + String(s.ax, 4);
    block += "," + String(s.ay, 4);
    block += "," + String(s.az, 4) + "\n";
  }
  f.print(block);
  buf.clear();
}

// En-tête GCSV — à écrire une seule fois en début de fichier

  "ascale,0.000598430\n"    // 1/16384 pour accéléro ±2g

4. Synchronisation temporelle : le nœud critique

La synchronisation entre le flux IMU et la vidéo est le problème le plus délicat de toute la chaîne. Un désalignement de 20 ms produit des artefacts visibles — la stabilisation corrige dans la mauvaise direction. Deux méthodes complémentaires :

Méthode 1 — Impulsion LED sur piste audio

Au démarrage de l'enregistrement, le M5StickC allume brièvement sa LED et émet simultanément un bip sonore sur son haut-parleur intégré. Le X-M5 enregistre le bip sur sa piste audio. En post-production, on identifie le pic audio (repère temporel vidéo) et on lit le timestamp correspondant dans le fichier GCSV. La différence constitue l'offset t0 à fournir à Gyroflow.

Appui bouton → LED flash + bip → timestamp GCSV noté
↕ synchronisation
Vidéo X-M5 → pic waveform audio → frame N identifiée

Méthode 2 — Sync automatique par corrélation

Gyroflow intègre un algorithme de synchronisation automatique basé sur la corrélation croisée entre le mouvement optique estimé dans la vidéo (optical flow dense) et les données gyroscopiques intégrées. Cette méthode est précise à ±1 frame (±33 ms à 30p) mais nécessite une vidéo avec des mouvements francs — elle échoue sur des plans très statiques.

En pratique, combiner les deux méthodes est optimal : la LED fournit un offset grossier (<500 ms), la corrélation affine à <5 ms.

5. Fixation mécanique et axes de référence

Le M5StickC doit être fixé rigidement sur le boîtier — tout jeu mécanique introduit une désynchronisation entre le mouvement du capteur et celui de l'objectif. On utilise une fixation griffe flash avec une embase en aluminium usinée ou imprimée en PETG (rigidité supérieure au PLA pour des raisons thermiques).

Le point crucial est la déclaration correcte du repère (frame) dans Gyroflow. Le M5StickC a son propre repère XYZ, qui ne coïncide pas forcément avec le repère caméra. Gyroflow demande l'orientation du gyroscope par rapport à la caméra via une matrice de rotation ou via des angles d'Euler (roll, pitch, yaw).

Axe caméra Axe MPU6886 (fixation griffe) Signe
Roulis (roll) Gz +1
Tangage (pitch) Gx −1 (axe inversé)
Lacet (yaw) Gy +1
Vérification empirique Pour valider le mapping d'axes : enregistrez quelques secondes en faisant un lacet pur (rotation horizontale uniquement). Chargez dans Gyroflow et observez quel axe est excité. Si Gx varie alors que vous faites un yaw, vos axes sont permutés. Cette vérification prend 5 minutes et évite des heures de débogage.

6. Pipeline de post-production

Étape 1 — Conversion et nettoyage du GCSV

Le script Python ci-dessous lit le fichier brut du M5StickC, applique une calibration de biais gyroscopique (offset mesuré à l'immobile), et exporte un GCSV propre compatible Gyroflow. 

#!/usr/bin/env python3
import pandas as pd
import numpy as np

GYRO_BIAS = np.array([0.42, -0.18, 0.07])  # dps — mesuré à plat, moteur off
SCALE    = 1 / 32.8                           # dps → rad/s

def clean_gcsv(path_in: str, path_out: str):
    df = pd.read_csv(path_in, comment='#',
                     names=['t','gx','gy','gz','ax','ay','az'],
                     skiprows=5)

    # Suppression des doublons et tri temporel
    df = df.drop_duplicates('t').sort_values('t').reset_index(drop=True)

    # Correction de biais
    df['gx'] -= GYRO_BIAS[0]
    df['gy'] -= GYRO_BIAS[1]
    df['gz'] -= GYRO_BIAS[2]

    # Rééchantillonnage à ODR constant (200 Hz) par interpolation linéaire
    t_uniform = np.arange(df['t'].iloc[0], df['t'].iloc[-1], 5000)
    for col in ['gx','gy','gz']:
        df[col] = np.interp(t_uniform, df['t'], df[col])
    df['t'] = t_uniform

    header = (
        "GYROFLOW IMU LOG\ntscale,1000\n"
        "gscale,1\nascale,1\n"
        "t,gx,gy,gz,ax,ay,az\n"
    )
    with open(path_out, 'w') as f:
        f.write(header)
        df.to_csv(f, index=False, header=False, float_format='%.5f')

clean_gcsv("raw_imu.gcsv", "cleaned_imu.gcsv")

Étape 2 — Calibration de l'objectif dans Gyroflow

Gyroflow nécessite un profil de calibration pour corriger la distorsion de l'objectif avant la stabilisation. Sans calibration, la stabilisation warp produit des artefacts en bordure de cadre. Pour le X-M5 avec le XF 18-55mm :

  1. Imprimer une grille de calibration Charuco (génération via OpenCV aruco.CharucoBoard)
  2. Filmer ~60 secondes en déplaçant lentement la caméra devant la grille — couvrir tous les bords
  3. Dans Gyroflow Toolbox, Lens Calibration → importer les frames → lancer l'optimisation
  4. Exporter le profil JSON et le placer dans ~/.gyroflow/lenses/

Étape 3 — Import dans Gyroflow et réglages

Une fois la vidéo et le GCSV importés, les paramètres critiques à ajuster :

Paramètre Gyroflow Valeur recommandée Explication
Smoothness 0.5 – 1.5 Plus élevé = horizons plus stables, moins de mouvement naturel
Horizon lock 100% si plan fixe Force l'horizontalité — désactiver pour plans dynamiques
Cropping mode Dynamic crop Préserve la FOV maximale au détriment de la taille de crop variable
Stabilization algo Default (quaternion) Bonne robustesse ; RollingShutter pour vidéo CMOS rapide
Rolling shutter correction ~16 ms pour X-M5 à 4K/30p À mesurer ou estimer depuis la fréquence de lignes du capteur
Rolling shutter du X-M5 Le capteur BSI CMOS du X-M5 présente un rolling shutter estimé entre 14 et 18 ms selon la résolution et le framerate. À 4K/30p, 16 ms est une bonne valeur de départ. Pour le mesurer précisément, filmer un stroboscope ou utiliser le test de la règle vibrante — la déformation de la droite verticale donne directement le temps de scan d'une frame.

Étape 4 — Export et intégration NLE

Gyroflow peut exporter directement en vidéo stabilisée (ProRes, H.264), ou fonctionner comme plugin dans DaVinci Resolve, Final Cut Pro et Premiere via le plugin officiel Gyroflow Toolbox. L'approche plugin est préférable : elle préserve la vidéo originale non destructive et permet d'ajuster les paramètres de stabilisation par clip sans réencoder. 

# Export CLI Gyroflow (headless, pour un batch de rushes)

7. Calibration du biais gyroscopique

Le MPU6886, comme tout MEMS, présente un biais en température et en vieillissement. Pour une calibration correcte, il faut mesurer l'offset à chaque session, idéalement après un temps de chauffe de 2 minutes (la température du die influence significativement le biais de zéro du gyroscope). 

// Calibration biais — à lancer au démarrage, caméra immobile 3 secondes
float biasBuf[3] = {0};
const int N_SAMPLES = 600;  // 3s à 200 Hz

for (int i = 0; i < N_SAMPLES; i++) {
  float gx, gy, gz;
  readGyro(&gx, &gy, &gz);  // lecture directe registres, pas FIFO
  biasBuf[0] += gx;
  biasBuf[1] += gy;
  biasBuf[2] += gz;
  delay(5);  // 200 Hz
}
biasBuf[0] /= N_SAMPLES;
biasBuf[1] /= N_SAMPLES;
biasBuf[2] /= N_SAMPLES;
// Stocker en EEPROM (NVS sur ESP32)
nvs.putFloat("bias_x", biasBuf[0]);
nvs.putFloat("bias_y", biasBuf[1]);
nvs.putFloat("bias_z", biasBuf[2]);

8. Limites de l'approche et pistes d'amélioration

Cette solution atteint des résultats très proches des systèmes intégrés (GoPro, Sony Catalyst) dans des conditions normales. Quelques limitations à connaître :

  • Vibrations haute fréquence : le DLPF à 41 Hz atténue les vibrations moteur, mais sur un drone ou un véhicule, des vibrations 60–200 Hz peuvent saturer le FIFO ou produire des artefacts. Augmenter le DLPF (DLPF_CFG=5 → 10 Hz BW) au détriment de la réactivité.
  • Jitter de l'ESP32 : l'ESP32 n'est pas un OS temps-réel. Des tâches WiFi ou Bluetooth actives peuvent introduire des délais dans la boucle de lecture. Désactiver WiFi/BT pendant l'enregistrement : WiFi.mode(WIFI_OFF); btStop();
  • Dérive thermique : sur des sessions longues (>20 min), la température du die monte et dérive le biais. Une recalibration mid-session (pause de 3 secondes à plat) peut être scriptée.
  • Précision du mapping d'axes : une erreur de quelques degrés dans la matrice de rotation produit une stabilisation imparfaite. Gyroflow permet d'affiner visuellement l'orientation avec le viewer 3D intégré.
